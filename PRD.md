# LawAgent 面试项目 PRD

## 目标与背景

LawAgent 是一个面向法律、合规、政策研究场景的 Agentic RAG 面试项目。它不是只演示“上传文档后聊天”的普通知识库 Demo，而是要完整展示一个候选人对企业级 RAG 项目的理解：数据治理、知识库构建、混合检索、多轮召回、自检改写、引用溯源、评测闭环、反馈迭代和可解释的工程取舍。

项目参考 `想法.md` 中的方向、B 站文章《我们用三个多月把公司近万份文档接进了大模型，最后学到的东西和想象完全不同》、三张流程/评测截图，以及法律 RAG、Agentic RAG、RAGAS、LangGraph、pgvector 等相关实践。核心定位是：做一个能被面试官追问并讲清楚细节的 AI Agent 项目，而不是套框架跑通的玩具项目。

本项目的第一版业务场景聚焦“法律/合规/外贸政策知识研究”。用户输入一个政策、合同、法规、合规风险或业务问题后，系统可以识别意图，路由到对应知识库或结构化数据工具，通过多轮检索和自检判断证据是否足够，最终给出带引用、带风险边界、可复核的回答。

## 产品目标

1. 建成一个可运行的法律/合规 Agentic RAG 系统，支持自然语言提问、意图识别、检索、生成、引用溯源和反馈。
2. 让 RAG Agent 具备多次召回能力：当证据不足、召回偏题或答案置信度低时，能自动改写问题、扩大召回、切换检索策略或明确拒答。
3. 建立数据治理流程：文档解析、清洗、元数据增强、分块、入库、图片/附件处理、版本记录和坏例回收。
4. 建立评测体系：先评检索层，再评生成层，最后评端到端体验，能用数据解释为什么这样设计。
5. 形成面试可讲述的项目资产：架构图、流程图、评测报告、Bad Case 分析、关键技术取舍和简历表达。

## 目标用户

1. 求职者本人：需要一个完整、能讲深、能放进简历的 AI Agent / RAG 项目。
2. 面试官：希望通过项目判断候选人是否真正理解 RAG 生产链路，而不是只会调用 LangChain 或向量库。
3. 模拟业务用户：法务、合规、外贸业务、政策研究人员，需要快速查询政策依据、合同条款、合规风险和相关制度。

## 核心场景

### 场景一：政策与法规问答

用户提出“某类产品出口到某地区需要满足哪些合规要求”或“某条法规对企业有什么影响”。系统需要识别问题属于政策/法规研究，检索对应法规、政策说明、内部解读和案例材料，输出结构化结论、依据来源、适用条件和不确定边界。

### 场景二：合同与条款风险分析

用户输入合同条款或上传合同片段，系统识别合同类型与关键条款，检索内部条款库、法规依据和风险规则，输出风险等级、触发原因、建议改写方向和引用来源。

### 场景三：跨知识库复杂问题

用户的问题同时涉及业务制度、法律法规和合规政策。Supervisor Agent 需要拆分问题并路由到多个子知识库，RAG Agent 对每个子问题分别召回，再汇总成统一答复。

### 场景四：结构化数据查询

用户询问“某政策命中了多少条历史案例”“某类风险在不同地区分布如何”等问题时，系统需要识别为结构化查询，调用 Text-to-SQL 链路访问 PostgreSQL 中的结构化表，返回统计结果，并与文本检索证据结合。

### 场景五：证据不足时的自检与拒答

当检索不到足够证据、证据冲突、引用来源不可信或问题超出知识库范围时，系统不能强行生成结论。它需要说明当前依据不足，给出已检索范围、缺失信息和建议补充材料。

## MVP 范围

第一版必须完成以下能力：

1. 文档导入与清洗：支持 Markdown、TXT、PDF 文本提取后的 Markdown 文档作为主要输入；Word、图片 OCR 可作为增强项。
2. 元数据增强：为每份文档生成摘要、关键词、可回答问题、领域标签、版本信息和来源路径。
3. 知识库分域：至少支持 3 个子知识库，建议为法规政策库、合同条款库、合规案例库；目标可扩展到 8 个子知识库。
4. 混合检索：支持向量检索和关键词检索，使用 RRF 或相似策略融合结果。
5. Agentic RAG：支持问题改写、多轮召回、证据充分性自检、答案忠实度检查和必要时拒答。
6. 多 Agent 编排：至少包含 Supervisor Agent、RAG Agent、Data Agent、SQL Agent 四类角色中的前三类；SQL Agent 可在第一版后半段接入。
7. 引用溯源：回答必须展示引用文档、片段、页码或 chunk 编号，用户能回看原始依据。
8. 评测集：构建黄金评测集，检索层标注正确 chunk，生成层标注参考答案。
9. 评测指标：检索层至少输出 Recall@K、MRR；生成层至少输出 Faithfulness、Answer Relevance、Context Precision、Context Recall。
10. 前端演示：提供一个可操作的聊天/研究界面，展示用户问题、改写问题、检索证据、最终回答、引用来源和反馈按钮。
11. 反馈闭环：记录用户点赞/点踩、点踩原因、检索结果、模型回答、耗时和 Bad Case 标签。

## 功能清单

### 数据治理模块

1. 文档上传或批量导入。
2. 文档清洗规则：去除页眉页脚、无意义时间戳、目录噪音、重复空行、无效链接、平台导出残留内容。
3. 文档结构化：保留标题层级、段落、表格文本、图片占位符和来源路径。
4. 元数据增强：调用大模型生成摘要、关键词、可回答问题、领域标签、适用地区、适用对象、文档版本。
5. 分块策略：短文档整篇入库；长文档按标题和语义切分；每个 chunk 保留父文档信息。
6. 父子文档关联：命中任一 chunk 后，可以回拉同一父文档的相邻或全部相关 chunk，避免“搜到摘要但漏掉正文答案”。
7. 图片和附件处理：将图片保存为可访问资源，文档中保留图片引用；MVP 可以先支持图片链接展示，OCR 作为后续增强。
8. 入库记录：记录清洗版本、Embedding 模型、向量维度、入库时间、失败原因和重试状态。

### 检索模块

1. 意图识别：判断用户问题属于法规政策、合同条款、合规案例、结构化统计、闲聊或越界问题。
2. 查询改写：结合对话历史，将指代性问题改写为完整问题。
3. 专业术语归一化：处理行业黑话、缩写、中英文混合表述和同义词。
4. 向量检索：基于 pgvector HNSW 索引召回语义相关片段。
5. 关键词检索：基于 PostgreSQL 全文检索、Elasticsearch 或 ParadeDB/BM25 召回精确关键词结果。
6. RRF 融合：将向量检索和关键词检索结果合并排序。
7. Rerank 精排：对候选片段做相关性重排；MVP 可先使用轻量交叉编码器或 LLM judge，后续再接专用 reranker。
8. 连坐召回：命中子 chunk 后按父文档补充上下文，提升具体条款、参数、例外条件的命中率。
9. 多轮召回：当召回不足时，自动尝试扩展 query、切换知识库、增加 TopK、引入同义词或拆分子问题。
10. 检索可解释：展示召回来源、分数、检索策略和是否经过改写。

### Agent 编排模块

1. Supervisor Agent：负责意图识别、任务拆解、知识库路由、工具选择和终止判断。
2. RAG Agent：负责查询改写、混合检索、多轮召回、证据自检、引用组织和回答生成。
3. Data Agent：负责文档清洗、元数据生成、入库任务、失败重试和数据质量报告。
4. SQL Agent：负责将自然语言转换为受限 SQL，仅查询白名单表和字段，返回结构化统计结果。
5. Self-Check 节点：检查证据是否足够、回答是否忠实于上下文、是否存在无法支持的结论。
6. Human-in-the-loop 节点：对高风险法律建议、低置信回答或用户点踩样本进入人工复核队列。

### 生成与回答模块

1. 输出结构固定为：结论、依据、适用条件、风险边界、建议动作、引用来源。
2. 回答必须基于检索证据，不允许编造法规、案例、条款编号或不存在的来源。
3. 对法律和合规问题必须带免责声明：系统提供研究辅助，不构成正式法律意见。
4. 对证据不足的问题要明确拒答或降级回答，并说明缺失材料。
5. 支持多轮追问，保留上下文，但每次检索前都要生成独立、完整的检索 query。
6. 支持展示引用卡片，包括文档名、来源、chunk 编号、摘要和命中原因。

### 评测与观测模块

1. 黄金评测集管理：每条样本包含问题、标准答案、正确 chunk、所属知识库、难度和问题类型。
2. 检索层评测：Recall@3、Recall@5、MRR@10、NDCG@10、Hit@3。
3. 生成层评测：Faithfulness、Answer Relevance、Context Precision、Context Recall。
4. 端到端指标：用户满意度、点踩率、转人工率、问题解决率、平均响应耗时。
5. Bad Case 分析：记录失败类型，例如未召回、召回但排序低、召回正确但生成错误、引用错误、拒答不合理。
6. 过程追踪：保存每次 Agent 运行的节点轨迹、检索参数、工具调用、耗时和最终决策。
7. 对比实验：支持比较不同 chunk 策略、TopK、Rerank 候选数、Embedding 模型和提示词版本。

### 前端演示模块

1. 聊天主界面：用户输入问题，系统返回带引用的法律/合规研究答复。
2. 检索过程侧栏：展示意图识别、改写 query、命中知识库、检索片段和自检结果。
3. 文档详情页：查看原始文档、清洗后 Markdown、chunk 列表、元数据和图片引用。
4. 评测看板：展示检索层、生成层和端到端指标。
5. 反馈入口：用户可以点赞、点踩并选择原因。
6. Bad Case 页面：按失败类型聚合问题，展示后续优化建议。

## 页面 / 工作流地图

### 页面清单

1. 问答工作台：核心演示页面。
2. 检索证据面板：展示 Agent 每一步如何检索、判断和生成。
3. 知识库管理页：导入文档、查看清洗结果、查看元数据和入库状态。
4. 文档详情页：查看原文、清洗版本、chunk、图片和引用历史。
5. 评测看板：查看 Recall、MRR、NDCG、RAGAS 指标、耗时和 Bad Case。
6. 反馈与复核页：查看用户点踩、人工复核状态和优化记录。

### 核心问答流程

1. 用户输入问题。
2. Supervisor Agent 判断问题类型和目标知识库。
3. Query Rewrite 节点结合历史对话改写为独立问题。
4. RAG Agent 同时执行向量检索和关键词检索。
5. 检索结果通过 RRF 融合和 Rerank 精排。
6. 证据自检节点判断是否足够回答。
7. 如果不足，进入第二轮召回：改写 query、扩大 TopK、补充父文档 chunk 或切换知识库。
8. 如果仍不足，输出拒答或补充材料建议。
9. 如果足够，生成带引用、带风险边界的回答。
10. 用户反馈被记录，用于 Bad Case 和后续评测集扩充。

### 数据入库流程

1. 批量导入法律、合规、政策、合同或案例文档。
2. 解析文档文本和图片占位。
3. 执行清洗规则并输出清洗报告。
4. 调用大模型生成摘要、关键词、可回答问题和领域标签。
5. 按分块策略生成 chunk，并保留父文档关系。
6. 生成 Embedding 并写入 PostgreSQL/pgvector。
7. 写入全文检索索引。
8. 运行小规模入库后评测，确认召回质量再扩大导入。

### 评测流程

1. 人工构建黄金评测集，先覆盖高频问题和复杂问题。
2. 对每条问题标注正确 chunk 和参考答案。
3. 先运行检索层评测，检索层未达标不进入生成层调优。
4. 检索层达标后运行生成层评测。
5. 对失败样本归因，分别调整清洗、分块、元数据、检索参数、Rerank 或提示词。
6. 每次优化后保存实验记录，形成面试可展示的评测报告。

## 数据范围

MVP 建议使用自建、可公开演示的数据，不接入真实客户隐私数据。

1. 法规政策库：公开法律法规、政策公告、监管要求。
2. 合同条款库：脱敏合同模板、常见条款、风险条款说明。
3. 合规案例库：公开案例摘要、行业解读、内部模拟案例。
4. 业务制度库：模拟公司制度、流程说明、审批规则。
5. 结构化数据表：政策地区、风险类型、条款分类、案例标签、反馈记录。

第一版建议目标数据规模：

1. 文档数量：100 到 150 份，覆盖至少 3 个子知识库。
2. 评测问题：不少于 60 条，其中 40 条用于检索层，20 条用于生成层。
3. 复杂问题：不少于 10 条，覆盖跨知识库、多跳推理、证据冲突和拒答。
4. 图片资料：至少 20 张，验证文档图片引用和前端展示能力。

## 验收标准

### 功能验收

1. 用户可以在问答界面提出法律/合规/政策问题，并得到结构化回答。
2. 每个非拒答回答都必须展示至少一个可点击或可定位的引用来源。
3. 用户追问“它”“这个政策”“刚才那个条款”等指代性问题时，系统能改写为完整 query 再检索。
4. 用户提出跨知识库问题时，Supervisor Agent 能展示路由结果，并调用多个知识库完成回答。
5. 当知识库无依据时，系统明确拒答，不编造法规、条款编号或案例。
6. 用户点踩后，系统记录原问题、答案、召回片段、点踩原因和时间。
7. 文档导入后可以查看清洗结果、元数据、chunk 和入库状态。
8. 评测看板可以展示最近一次评测的检索层和生成层指标。

### 检索质量验收

1. 黄金评测集上 Recall@3 达到 0.85 以上。
2. 黄金评测集上 Recall@5 达到 0.90 以上。
3. MRR@10 达到 0.85 以上。
4. Hit@3 达到 0.90 以上。
5. 每次检索能记录检索策略、TopK、Rerank 输入数量和最终命中文档。

如果第一版数据较少，可以先把指标作为目标阈值，同时在报告中说明样本量、失败样本和下一轮优化方向。

### 生成质量验收

1. Faithfulness 达到 0.80 以上。
2. Answer Relevance 达到 0.80 以上。
3. Context Precision 达到 0.75 以上。
4. Context Recall 达到 0.75 以上。
5. 人工抽检 20 条回答，不允许出现无引用的法律结论。
6. 高风险问题必须带风险边界和“非正式法律意见”说明。

### 工程验收

1. 后端提供清晰的 API：问答、文档导入、评测运行、反馈提交、文档详情。
2. Agent 每个节点都有日志和 trace，能复盘一次回答为什么这么生成。
3. 数据库记录会话、消息、反馈、文档、chunk、评测结果和运行轨迹。
4. 关键配置可调整：TopK、Rerank 候选数、是否启用连坐召回、是否启用二次召回。
5. 本地开发环境可通过 README 中的命令启动。
6. 项目附带演示问题集和面试讲解材料。

## 非目标

1. MVP 不提供正式法律咨询服务，不承诺法律意见的最终正确性。
2. MVP 不处理真实客户敏感合同、案件材料或商业秘密。
3. MVP 不实现完整权限系统，只保留权限设计说明和基础角色隔离。
4. MVP 不追求超大规模文档量，重点展示完整链路和可解释优化过程。
5. MVP 不做复杂工作流审批、企业 SSO、计费、租户管理。
6. MVP 不一开始接入过多模型供应商，优先保证链路稳定和可复盘。
7. MVP 不把所有 MCP 工具做成亮点；工具调用应服务于 Data Agent、SQL Agent 和可观测性，而不是堆概念。

## 技术方向

### 推荐技术栈

1. 后端：Python、FastAPI。
2. Agent 编排：LangGraph，用显式状态图表达 Supervisor、RAG、Data、SQL、自检节点。
3. 数据库：PostgreSQL。
4. 向量检索：pgvector，使用 HNSW 索引。
5. 关键词检索：PostgreSQL 全文检索、Elasticsearch 或 ParadeDB/BM25，MVP 优先选择最少依赖的方案。
6. 检索融合：RRF。
7. 前端：Next.js 或 React，做问答工作台、证据面板和评测看板。
8. 评测：自建脚本加 RAGAS 指标。
9. 文档解析：优先支持 Markdown/TXT/PDF 文本；Word、OCR、图片理解作为增强。
10. 对象存储：MVP 可用本地静态目录模拟，后续替换为 MinIO。

### 技术取舍原则

1. 面试项目优先展示“为什么这样设计”，不盲目堆中间件。
2. 向量库优先选择 PostgreSQL/pgvector，减少系统复杂度；当数据量和并发证明不够时，再讨论 Milvus。
3. 第一版可以先不接 Elasticsearch，用 PostgreSQL 全文检索实现关键词召回；如果要突出企业级检索，可以在第二阶段引入 Elasticsearch。
4. Rerank 候选数必须通过评测确定，不能默认越大越好。
5. Agentic RAG 的循环次数必须有上限，避免无限自检和成本失控。
6. SQL Agent 必须使用白名单、只读权限和查询限制，不能让模型自由拼接任意 SQL。

## 数据模型草案

### documents

保存父文档信息：标题、来源、领域、版本、摘要、关键词、可回答问题、清洗状态、创建时间。

### chunks

保存文档片段：父文档 ID、chunk_index、标题路径、正文、embedding、token 数、图片引用、元数据。

### conversations

保存会话：用户、标题、创建时间、最近更新时间。

### messages

保存问答消息：原始问题、改写问题、答案、引用、模型、耗时、状态。

### retrieval_traces

保存检索轨迹：query、知识库、检索策略、TopK、候选结果、分数、Rerank 结果、自检结论。

### feedback

保存用户反馈：message_id、点赞/点踩、原因、人工备注、是否进入评测集。

### eval_cases

保存评测样本：问题、标准答案、正确 chunk、知识库、难度、标签。

### eval_runs

保存评测运行结果：指标、参数、失败样本、运行时间、版本。

## 安全和数据边界

1. 所有演示数据必须来自公开资料、脱敏材料或自造模拟材料。
2. 回答中不得输出“确定胜诉”“必然合规”等绝对化法律结论。
3. 高风险问题必须提示用户咨询专业律师或合规负责人。
4. SQL Agent 只允许查询白名单表，只读，不允许执行写入、删除、DDL。
5. 文档导入日志不能记录 API Key、密钥、真实个人身份信息。
6. 用户反馈和会话数据需要可删除，便于演示隐私处理意识。
7. Prompt 中必须要求模型基于引用回答，禁止无依据补全。

## 非功能需求

1. 本地演示环境单次普通问答响应时间目标小于 10 秒。
2. 启用二次召回或 Rerank 后，单次复杂问答目标小于 20 秒。
3. 检索失败、模型调用失败、SQL 生成失败时，前端展示明确错误而不是空白。
4. 数据导入任务支持断点续跑，单份失败不影响整体批次。
5. 所有关键链路有日志，便于面试时现场定位问题。
6. 配置集中管理，避免把模型 Key、数据库地址硬编码进代码。
7. 核心链路有单元测试或脚本级验证：清洗、分块、检索、RRF、评测指标。

## 面试展示重点

1. 数据清洗不是附属工作，而是 RAG 质量的主要来源。
2. 分块策略要结合文档长度和业务结构，不能只套固定 chunk_size。
3. 元数据增强能提高用户问法和文档内容之间的匹配能力。
4. 混合检索解决“语义相关”和“关键词精确命中”的互补问题。
5. 连坐召回解决“命中父文档但漏掉具体答案 chunk”的问题。
6. Rerank 候选数要用 Recall 和耗时数据决策。
7. Agentic RAG 的价值是让系统能自检证据是否足够，并在不足时多轮召回或拒答。
8. 法律场景必须强调依据可追溯、结论有边界、风险可解释。
9. 评测要分层：检索层未达标时，不应盲目调 Prompt。
10. 反馈闭环要从 MVP 阶段接入，因为点踩就是 Bad Case 标注。

## 风险与依赖

1. 数据集质量风险：如果文档太少或太干净，无法体现真实 RAG 难点。应准备噪音文档、长文档、表格、图片和跨库问题。
2. 法律正确性风险：模型可能输出过度确定的法律结论。必须用引用、边界、拒答和人工复核降低风险。
3. 评测样本不足风险：样本太少会让指标不可信。第一版至少 60 条，后续逐步扩充。
4. 工程复杂度风险：多 Agent、SQL、Rerank、评测、前端都做可能失控。第一阶段先打通一条完整闭环。
5. 成本风险：多轮召回和 LLM judge 会增加调用成本。需要限制最大循环次数和评测运行频率。
6. 技术堆叠风险：如果同时接 Elasticsearch、Milvus、MinIO、K8s，会偏离面试项目主线。MVP 优先用 PostgreSQL/pgvector 和本地文件模拟。

## 第一阶段实现切片

第一阶段目标是跑通“可评测的法律/合规 Agentic RAG 闭环”，不追求所有增强功能。

### 阶段一范围

1. 准备 30 到 50 份公开或模拟法律/合规文档。
2. 建立文档导入、清洗、元数据生成和 chunk 入库流程。
3. 使用 PostgreSQL/pgvector 建立向量检索。
4. 实现基础关键词检索和 RRF 融合。
5. 实现 Query Rewrite、一次自检和最多一次二次召回。
6. 实现回答生成和引用展示。
7. 建立 20 条黄金评测问题，输出 Recall@K 和 MRR。
8. 实现最小前端问答界面和检索证据展示。
9. 记录用户反馈和一次完整 retrieval trace。

### 阶段一验收

1. 用户能提问并看到带引用回答。
2. 系统能展示改写后的 query、检索结果和最终引用。
3. 至少 20 条评测问题能自动跑完并输出指标。
4. 至少 3 个 Bad Case 能被归因到具体类别。
5. README 中有启动命令、演示问题和项目讲解路径。

## 第二阶段增强方向

1. 扩展到 100 到 150 份文档和 60 条以上评测集。
2. 引入 RAGAS 生成层评测。
3. 引入 Rerank 并做候选数、耗时、Recall 的对比实验。
4. 加入 SQL Agent，支持结构化数据查询。
5. 增加图片引用展示和 OCR。
6. 增加 Bad Case 看板和人工复核流程。
7. 输出面试材料：架构图、流程图、评测报告、简历项目描述和追问回答。

## 建议验证方式

1. 数据导入验证：随机抽 5 份文档，人工检查清洗后内容、元数据和 chunk 是否合理。
2. 检索验证：运行黄金评测集，查看 Recall@3、Recall@5、MRR@10。
3. 生成验证：人工抽检 20 条回答，确认结论是否由引用支持。
4. 自检验证：构造 5 条知识库外问题，确认系统拒答或要求补充材料。
5. 跨库验证：构造 5 条跨法规、合同、案例的问题，确认 Supervisor 能路由到多个知识库。
6. 反馈验证：点踩一条回答，确认反馈、检索轨迹和 Bad Case 标签被记录。
7. 面试验证：按“背景 -> 数据治理 -> 检索策略 -> Agentic RAG -> 评测 -> Bad Case -> 下一步优化”的顺序做 5 分钟项目讲解。
