# LawAgent

LawAgent 是一个面向法律合规场景的可追溯研究助手。这个上下文用于统一项目里的领域词，避免把研究辅助、法律意见、合同审查和通用问答混在一起。

## Language

**法律合规场景**:
用户围绕法规、政策、合同条款、合规风险和内部制度提出问题，需要系统整理依据、解释适用边界并提示风险。
_Avoid_: 通用法律问答, 外贸系统, AI 律师

**可追溯研究助手**:
帮助用户基于可引用材料完成法律合规问题的检索、依据整理和初步分析，但不替代律师或合规负责人给出最终意见。
_Avoid_: 法律咨询机器人, 自动律师, 最终裁判

**法规政策研究**:
第一版优先支持的任务类型，围绕公开法规、监管政策、合规要求和政策解读整理适用依据、条件、风险边界和行动建议。
_Avoid_: 合同审查主线, 企业制度问答主线, 通用问答

**企业数据合规政策研究**:
第一版法规政策研究的具体切口，围绕个人信息处理、数据共享、数据出境、隐私告知和企业数据处理义务整理合规依据。
_Avoid_: 外贸合规主线, 劳动用工合规主线, 全领域法律合规

**辅助英文隐私语料**:
用于增加文档规模、清洗难度和评测样本的公开隐私政策/隐私问答数据，不作为第一版产品的多语言法律研究承诺。
_Avoid_: 中英文法律研究主线, GDPR 专家系统

**数据治理流水线**:
将原始法规、政策、隐私政策和模拟制度材料加工成可检索知识库的四阶段流程，包括原始采集、清洗规范、语义增强和检索入库。
_Avoid_: 直接向量化, 手工整理干净文本

**清洗前后对比**:
第一版必须展示的证据，用于说明原始材料中的噪声如何被识别、移除或结构化，并保留可复核的清洗结果。
_Avoid_: 只展示最终 chunk, 不可解释清洗

**Agentic Workflow**:
第一版采用的在线问答形态，用显式工作流节点完成意图识别、查询改写、混合检索、证据自检、二次召回、引用校验和拒答，而不是让多个自治 Agent 自由对话。
_Avoid_: 真多 Agent, 角色扮演式 Agent 协作, 黑盒工具调用

**二次召回**:
LawAgent 第一版必须突出的 Agentic RAG 能力，指系统在首次检索后判断证据不足时，受控地改写查询、扩大召回或切换检索策略，再执行一次补充检索。
_Avoid_: 无限循环检索, 单次普通 RAG, 无条件扩大 TopK

**证据自检**:
回答生成前的判断步骤，用于检查检索证据是否足够、是否命中正确主题、是否支持用户问题，并决定直接回答、二次召回或拒答。
_Avoid_: 生成后才补引用, 无依据生成

**高精度混合检索**:
LawAgent 第一版的检索主线，结合向量检索、关键词检索、RRF 融合、可选精排和父文档补充，目标是提高法规政策问题的正确依据命中率。
_Avoid_: 只做向量检索, 只做关键词搜索, 黑盒知识库检索

**双检索存储**:
第一版采用 PostgreSQL/pgvector 与 Elasticsearch 的组合：PostgreSQL 保存主数据、元数据、trace 和向量索引，Elasticsearch 承担关键词/BM25 检索与后续搜索调优。
_Avoid_: 纯 PostgreSQL 检索, 第一版引入 Milvus, 只用 Elasticsearch 承担所有业务数据

**中等规模黄金评测集**:
第一版用于证明检索、生成和 Agentic 能力的评测样本集合，规模不追求大，但必须覆盖高频问题、复杂问题、二次召回、拒答和证据冲突。
_Avoid_: 只做人工随问随测, 过小演示题集, 无标注评测

**法律合规研究工作台**:
LawAgent 第一版前端主体验，以用户完成法律合规问题研究为中心，突出提问、研究结论、依据引用、风险边界和可继续追问的工作流。
_Avoid_: 普通聊天机器人界面, 纯管理后台, RAG 调试台

**用户可见依据过程**:
前端可以展示引用来源、检索依据和必要的推理过程，但不展示文档清洗、chunk 生成、索引入库等后台数据治理细节。
_Avoid_: 清洗前后对比界面, chunk 调试界面, 让用户理解底层 RAG 流程

**数据治理优先实现**:
第一阶段先建设数据采集、清洗、语义增强、分块、入库和评测数据准备，不急于完成前端产品演示。
_Avoid_: 先做聊天 Demo, 用干净小样本绕过数据处理

**全国人大法律法规数据库**:
LawAgent 中文法律法规材料的首选权威来源，使用 `flk.npc.gov.cn` 及其公开 Word/PDF 资源获取法律、行政法规、地方性法规和司法解释文本。
_Avoid_: 以转载站点作为主来源, 来源不明的法规合集
