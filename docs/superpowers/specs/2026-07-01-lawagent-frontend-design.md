# LawAgent 前端设计方案

## 设计目标

LawAgent 第一版前端定位为法律合规研究/审查工作台，帮助用户围绕企业数据合规问题完成提问、提交待审查材料、查看研究结论、核对依据、理解风险边界、继续追问和导出研究摘要。

当前 mockup 中的研究结论、建议动作、风险边界和依据卷宗方向正确，但它仍偏向“用户问一个法规问题，系统给研究结论”。后续实现需要把主闭环升级为“用户提交自己的材料，系统对具体业务场景做合规审查”：用户上传或粘贴隐私政策、PDF 文本、数据处理协议、SDK 清单、字段表或业务说明后，系统抽取材料事实，匹配法规依据，输出风险等级、缺失信息、建议动作和引用来源。

前端不承担文档清洗、chunk 调试、embedding 入库、索引构建等后台数据治理展示。数据治理仍是项目能力和面试讲解重点，但普通用户界面只展示对研究任务有直接价值的信息。

## 设计原则

1. 用户第一屏看到的是可直接使用的研究工具，而不是 RAG 调试台。
2. 每个结论必须有引用来源和风险边界。
3. Agentic RAG 过程只以可展开的依据过程呈现，不打断用户完成研究任务。
4. 视觉风格要冷静、专业、可审计，避免 AI 聊天玩具感和紫色渐变风格。
5. 页面数量保持克制，第一版只服务核心演示闭环。

## 信息架构

第一版包含三个用户可见区域：

1. 研究工作台
2. 历史研究
3. 评测看板

研究工作台是主入口，但它不应只是问答框。主入口需要同时承载“输入研究问题”和“提交待审查材料”两个动作。历史研究用于回看过往问题、材料摘要、结论和引用。评测看板用于项目展示和面试说明，不是普通用户日常主流程。

不做独立页面：

1. 文档治理页
2. 知识库管理后台
3. chunk 调试页
4. Prompt 调试页
5. 多用户权限后台

## 主页面：法律合规研究工作台

### 页面结构

主页面采用三栏结构：

1. 左侧导航与常用场景
2. 中间研究案卷
3. 右侧依据卷宗

左侧导航包含：

1. LawAgent 品牌区
2. 研究工作台
3. 历史研究
4. 评测看板
5. 常用场景入口

中间研究案卷包含：

1. 研究问题输入框
2. 待审查材料输入区，支持粘贴文本和后续文件上传
3. 开始研究/开始审查按钮
4. 材料事实摘要
5. 研究结论
6. 适用依据摘要
7. 建议动作
8. 风险边界
9. 继续追问
10. 生成研究摘要
11. 反馈问题

右侧依据卷宗包含：

1. 证据充分状态
2. 是否触发补充检索
3. 引用来源列表
4. 来源类型和权威级别
5. 可展开依据过程

### 回答结构

每次研究回答固定展示：

1. 研究结论
2. 材料事实摘要
3. 适用依据
4. 风险等级和触发原因
5. 建议动作
6. 风险边界
7. 引用来源

法律合规结论不能只显示一段自然语言回答。必须让用户能区分“系统判断”“依据材料”“建议动作”和“仍需人工复核的边界”。

材料事实摘要用于解释系统如何理解用户提交的业务场景，例如数据类型、处理目的、接收方、是否出境、是否涉及敏感个人信息、授权方式和缺失信息。没有材料时，页面可以退化为法规研究问答；有材料时，回答必须体现材料对风险判断的影响。

### 依据过程

依据过程默认折叠或弱化展示。可展示的信息包括：

1. 查询是否被改写
2. 是否使用混合检索
3. 是否触发一次补充检索
4. 证据自检结果
5. 最终依据状态

不展示的信息包括：

1. 清洗前后文本
2. chunk 列表
3. embedding 向量
4. 索引构建状态
5. 后台数据治理流水线

## 历史研究

历史研究用于保存和回看用户曾经完成的研究案卷。

每条历史记录包含：

1. 原始问题
2. 生成时间
3. 研究结论摘要
4. 证据状态
5. 引用来源数量
6. 是否触发补充检索

用户可以打开历史研究继续追问，也可以导出摘要。

第一版历史研究可以先以本地会话或数据库记录为主，不需要复杂搜索、筛选和团队协作。

## 评测看板

评测看板用于项目展示和面试讲解。它解释系统为什么不是普通 RAG。

第一版展示：

1. pgvector 向量检索 baseline
2. Elasticsearch BM25 baseline
3. Hybrid Retrieval
4. Hybrid + Agentic RAG
5. Recall@3
6. Recall@5
7. MRR@10
8. 二次召回提升
9. 拒答准确率
10. Bad Case 列表

评测看板不面向普通业务用户。它面向项目作者、面试官和技术评审。

## 视觉系统

设计系统基于 `design-system/lawagent/MASTER.md`，但在实现时需要偏产品工具界面，而不是传统律所官网。

### 色彩

核心色彩：

1. 权威蓝：`#1E3A8A`
2. 深墨色：`#0F172A`
3. 背景灰白：`#F8FAFC`
4. 边框灰：`#CBD5E1`
5. 金色强调：`#B45309`
6. 成功绿：`#047857`
7. 风险红：`#B91C1C`

使用规则：

1. 蓝色用于主导航、主按钮、权威状态。
2. 金色用于注意事项、风险边界、需要人工复核的提示。
3. 绿色用于证据充分、已完成、已命中。
4. 红色只用于错误、拒答或严重风险。
5. 不使用紫色、粉色和大面积渐变。

### 字体

建议字体：

1. 标题：`Georgia` 或 `Noto Serif SC`，体现法律研究和文档感。
2. 正文：`Lato`、`Noto Sans SC` 或系统无衬线字体，保证长文本可读。

实现时可以使用在线字体，也可以先用系统字体兜底，避免字体加载失败影响演示。

### 布局密度

LawAgent 是工作型产品，不做营销首页式大留白。界面需要信息密度适中：

1. 桌面端使用三栏布局。
2. 平板端隐藏或折叠右侧依据卷宗。
3. 移动端改为单列，依据来源放在回答下方。
4. 固定格式区域必须有稳定尺寸，避免加载和状态变化造成布局跳动。

## 交互状态

### 空状态

用户还没有提问时，主区域显示：

1. 一句话说明系统用途
2. 三到四个常用场景问题
3. 研究问题输入框
4. 待审查材料粘贴/上传入口
5. 开始研究或开始审查按钮

### 研究中

用户提交问题后，界面显示：

1. 输入框禁用或显示加载状态
2. 中间区域展示研究进度
3. 右侧依据卷宗显示“正在检索依据”
4. 不显示空白页面

### 材料审查中

用户提交待审查材料后，界面显示：

1. 材料解析状态
2. 已抽取事实和缺失事实
3. 当前匹配到的法规依据数量
4. 是否需要补充检索或补充材料
5. 最终风险结论生成进度

### 依据不足

证据自检不足时，界面显示：

1. 已检索范围
2. 缺失信息
3. 建议用户补充的问题
4. 明确说明不能形成强结论

### 拒答

知识库外或证据不足的问题，系统应显示：

1. 无法回答的原因
2. 已尝试的检索范围
3. 可补充的材料
4. 不生成无依据法律结论

### 错误状态

模型调用、检索服务、评测服务失败时，界面必须显示明确错误原因和可重试动作。

## 可访问性要求

1. 普通正文对比度不低于 4.5:1。
2. 所有按钮和可点击卡片有可见 hover 与 focus 状态。
3. 所有交互目标不小于 44px 高度。
4. 表单输入必须有可见标签或可访问名称。
5. 不只用颜色表达状态，状态文字必须同步展示。
6. 动画只使用 150-300ms 的轻量过渡，并支持 `prefers-reduced-motion`。
7. 移动端不能出现横向滚动。

## 响应式规则

桌面端大于等于 1200px：

1. 三栏布局完整展示。
2. 左侧导航 240px 左右。
3. 右侧依据卷宗 360-400px。

平板端 768px 到 1199px：

1. 左侧导航压缩为图标或窄栏。
2. 右侧依据卷宗折叠为抽屉。

移动端小于 768px：

1. 单列布局。
2. 导航收进顶部或底部菜单。
3. 依据来源放在回答下方。
4. 评测表格改为卡片列表。

## 验收标准

1. 用户可以输入企业数据合规问题并看到结构化研究回答。
2. 用户可以粘贴或上传待审查材料，并看到基于材料事实生成的风险结论。
3. 每个非拒答回答都展示引用来源和风险边界。
4. 有材料输入时，页面必须展示系统抽取的关键事实和缺失信息。
5. 用户能看到系统是否触发补充检索，但不会看到文档清洗、chunk 或 embedding 细节。
6. 历史研究可以回看过往问题、材料摘要和结论摘要。
7. 评测看板可以展示 baseline 对比和 Agentic RAG 增益。
8. 页面在 375px、768px、1024px、1440px 宽度下不出现内容重叠或横向滚动。
9. 加载、空状态、拒答、错误状态都有清晰用户反馈。
10. 视觉风格符合专业、克制、可审计的法律合规产品气质。

## 当前确认结果

已确认：

1. 不做文档清洗前端页面。
2. 前端主要服务用户直接使用。
3. 主体验是法律合规研究工作台。
4. 依据过程可以展示，但只展示用户能理解的证据状态。
5. 高保真 mockup 当前未发现问题。

未进入第一版：

1. 知识库管理后台。
2. 文档清洗可视化。
3. Prompt 调试器。
4. 多用户权限系统。
5. 在线文档编辑器。
