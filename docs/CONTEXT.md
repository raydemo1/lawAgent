# LawAgent

LawAgent 是一个面向法律合规场景的可追溯研究助手。这个上下文用于统一项目里的领域词，避免把研究辅助、法律意见、合同审查和通用问答混在一起。

## Language

**法律合规场景**:
用户围绕法规、政策、合同条款、合规风险和内部制度提出问题，需要系统整理依据、解释适用边界并提示风险。
_Avoid_: 通用法律问答, 外贸系统, AI 律师

**可追溯研究助手**:
帮助用户基于可引用材料完成法律合规问题的检索、依据整理和初步分析，但不替代律师或合规负责人给出最终意见。
_Avoid_: 法律咨询机器人, 自动律师, 最终裁判

**法规政策研究**:
第一版优先支持的任务类型，围绕公开法规、监管政策、合规要求和政策解读整理适用依据、条件、风险边界和行动建议。
_Avoid_: 合同审查主线, 企业制度问答主线, 通用问答

**企业数据合规政策研究**:
第一版法规政策研究的具体切口，围绕个人信息处理、数据共享、数据出境、隐私告知和企业数据处理义务整理合规依据。
_Avoid_: 外贸合规主线, 劳动用工合规主线, 全领域法律合规

**企业数据合规审查人**:
LawAgent 第一版的主用户，指企业内部负责数据出境、个人信息处理、隐私政策、第三方共享和行业/地区适用性审查的法务或合规人员。
_Avoid_: 普通员工, 外部律师客户, 只看技术指标的面试官

**材料驱动的合规审查**:
LawAgent 后续产品闭环的目标形态。用户不仅提出法规问题，还会上传或粘贴隐私政策、PDF、数据处理协议、SDK 清单、业务说明、字段表等材料；系统先抽取具体业务事实，再匹配法规和政策依据，输出风险等级、触发原因、缺失信息、建议动作和引用来源。
_Avoid_: 只问法规概念, 只生成泛泛研究回答, 不结合用户材料判断具体场景

**审查事实**:
材料驱动合规审查中从用户材料抽取出的固定字段事实，包括业务活动、数据类型、敏感个人信息、是否出境、境外接收方、处理目的、同意或授权依据、行业、地区和缺失信息。
_Avoid_: 自由文本材料摘要, 无结构问答上下文, 只靠模型临场判断

**上传文档解析器**:
材料驱动合规审查的入口能力。第一版默认采用 Docling 解析用户上传 PDF/DOCX/HTML/图片型文档，将标题、段落、表格、页码和结构化输出映射为 LawAgent 的统一 Document；MinerU/DeepDoc 只在扫描件、复杂表格或 Docling 效果不足时作为增强候选。
_Avoid_: 只做纯文本粘贴, 一次性接入所有重型 parser, 把解析器替代法律元数据和证据治理

**辅助英文隐私语料**:
用于增加文档规模、清洗难度和评测样本的公开隐私政策/隐私问答数据，不作为第一版产品的多语言法律研究承诺。
_Avoid_: 中英文法律研究主线, GDPR 专家系统

**数据治理流水线**:
将原始法规、政策、隐私政策和模拟制度材料加工成可检索知识库的四阶段流程，包括原始采集、清洗规范、语义增强和检索入库。
_Avoid_: 直接向量化, 手工整理干净文本

**清洗前后对比**:
第一版必须展示的证据，用于说明原始材料中的噪声如何被识别、移除或结构化，并保留可复核的清洗结果。
_Avoid_: 只展示最终 chunk, 不可解释清洗

**Agentic Workflow**:
采用确定性 Supervisor 的在线审查形态，用显式工作流节点完成事实抽取、问题规划、混合检索、证据自检、二次召回、结果审查、引用校验和拒答。复杂场景可以启用有界 Multi-Agent：Case Analyst 生成议题和受控研究查询，Evidence Researchers 按议题融合证据，Compliance Reviewer 生成初始结果，Evidence Critic 只在高风险、证据不足或已补检索场景输出结构化修订动作；最多一次定向补检索和一次局部 patch 修订，不允许多个自治 Agent 自由对话。
_Avoid_: 无限 Agent 对话, 角色扮演式协作, 黑盒工具调用, 无终止条件的反思循环

**LLM-owned Review Workflow**:
材料驱动合规审查的 LLM 主导形态。审查事实抽取、query planning、结构化审查结果和条件式 Evidence Critic 由 LLM 节点负责；证据自检先判断可验证的确定性条件，只有库外范围或事实过少等语义歧义才调用 LLM。程序负责 schema 校验、引用门禁、trace、检索后端调用、Critic 触发条件、单次修订上限和错误暴露。当前实现先只接入 DeepSeek，不引入多 provider 能力字段；每个 LLM 节点 prompt 必须给出目标 JSON 示例，LLM 输出按 JSON prompt 约束后直接走 Pydantic 严格校验和节点级 retry。初始 Reviewer 或可选 Revision 失败时，Multi-Agent 可以保留程序生成或原先已经验证的可用结果，但必须标记为 degraded success；无法产生可信结果时才是 hard failure。不要再拆出独立 risk classifier、missing info detector 或 recommendation generator。
_Avoid_: 不可观察的规则 fallback, 模型失败后伪装成 clean success, 把规则判断混进模型输出

**二次召回**:
LawAgent 第一版必须突出的 Agentic RAG 能力，指系统在首次检索后判断证据不足时，受控地改写查询、扩大召回或切换检索策略，再执行一次补充检索。
_Avoid_: 无限循环检索, 单次普通 RAG, 无条件扩大 TopK

**证据自检**:
回答生成前的判断步骤，用于检查检索证据是否足够、是否命中正确主题、是否支持用户问题，并决定直接回答、二次召回或拒答。
_Avoid_: 生成后才补引用, 无依据生成

**证据不足审查结果**:
LLM-owned Review Workflow 正常完成后给出的业务结论，表示当前材料、检索证据或二次召回后的证据仍不足以支持确定审查判断，需要用户补充事实或依据。它是有效的 ReviewResult，而不是系统故障。
_Avoid_: 模型调用失败, schema 校验失败, 检索服务不可用

**审查流程失败**:
系统无法产出可信 ReviewResult 的故障状态，例如 LLM 多次输出不合格、模型 API 超时、检索后端不可用、引用校验重试耗尽。它必须显式暴露给用户和 trace，不能伪装成证据不足审查结果。若 workflow 已经开始，节点级失败应以结构化 `review_failed` 响应返回，而不是仅以 HTTP 500/502 中断。`review_failed` 响应只保留最小字段：状态、失败节点、失败原因、用户可读说明、尝试次数和 trace id；raw output、validation errors、token usage、latency 等细节放入后端 trace。
_Avoid_: 业务证据不足, 用户材料缺信息, 合法拒答

**高精度混合检索**:
LawAgent 第一版的检索主线，结合向量检索、关键词检索、RRF 融合、可选精排和父文档补充，目标是提高法规政策问题的正确依据命中率。
_Avoid_: 只做向量检索, 只做关键词搜索, 黑盒知识库检索

**双检索存储**:
第一版采用 PostgreSQL/pgvector 与 Elasticsearch 的组合：PostgreSQL 保存主数据、元数据、trace 和向量索引，Elasticsearch 承担关键词/BM25 检索与后续搜索调优。
_Avoid_: 纯 PostgreSQL 检索, 第一版引入 Milvus, 只用 Elasticsearch 承担所有业务数据

**中等规模黄金评测集**:
第一版用于证明检索、生成和 Agentic 能力的评测样本集合，规模不追求大，但必须覆盖高频问题、复杂问题、二次召回、拒答和证据冲突。
_Avoid_: 只做人工随问随测, 过小演示题集, 无标注评测

**法律合规研究工作台**:
LawAgent 第一版前端主体验，以用户完成法律合规问题研究为中心，突出提问、材料输入、研究结论、依据引用、风险边界和可继续追问的工作流。现有问答式研究是起点，后续应升级为材料驱动的合规审查，而不是停留在“用户问一个法规问题，系统给研究结论”。
_Avoid_: 普通聊天机器人界面, 纯管理后台, RAG 调试台

**用户可见依据过程**:
前端可以展示引用来源、检索依据和必要的推理过程，但不展示文档清洗、chunk 生成、索引入库等后台数据治理细节。
_Avoid_: 清洗前后对比界面, chunk 调试界面, 让用户理解底层 RAG 流程

**数据治理优先实现**:
第一阶段先建设数据采集、清洗、语义增强、分块、入库和评测数据准备，不急于完成前端产品演示。
_Avoid_: 先做聊天 Demo, 用干净小样本绕过数据处理

**全国人大法律法规数据库**:
LawAgent 中文法律法规材料的首选权威来源，使用 `flk.npc.gov.cn` 及其公开 Word/PDF 资源获取法律、行政法规、地方性法规和司法解释文本。
_Avoid_: 以转载站点作为主来源, 来源不明的法规合集
